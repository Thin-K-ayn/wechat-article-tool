# 长内容拆系列文章工作流 SOP

适用范围：
- 长视频
- 课程讲义
- 长论文
- 长播客 / 长访谈
- 任何不适合直接压成一篇公众号文章的长内容

目标：
- 先形成稳定拆分方案
- 再按拆分点逐篇完成
- 让第二天继续时不会丢失上下文

## 什么时候启用这套 SOP

满足任意一条就启用：
- 单个来源明显过长，不适合直接写成一篇公众号文
- 内容内部天然包含多个主题层次
- 用户明确希望拆成系列
- 预计需要跨多天完成

## 标准流程

### 1. 快速通读原始内容

目标：
- 不是逐字精读
- 而是先抓主线、章节和概念层级

输出：
- 这份内容主要在讲什么
- 核心主线有几条
- 哪些部分适合拆开写

### 2. 决定拆成几篇

拆分原则：
- 每篇只讲一条主线
- 每篇长度适合公众号阅读
- 拆分边界按“问题”而不是按“时间轴”
- 尽量让每篇都能独立成立

常见拆法：
- 概念篇 / 框架篇 / 原理篇
- 背景篇 / 方法篇 / 应用篇
- 入门篇 / 进阶篇 / 实战篇

输出：
- 建议拆成几篇
- 每篇各自解决什么问题

### 3. 建立系列计划文件

必须落一个独立 Markdown 文件，放在：
- `./plans/`

建议命名：
- `YYYY-MM-DD-<topic>-series-plan.md`

计划文件必须包含：
- 来源链接
- 系列目标
- 拆分原则
- 总进度勾选框
- 每篇的状态
- 每篇的主题
- 每篇要回答的问题
- 每篇的核心内容
- 每篇的标题方向
- 每篇完成后的记录栏位

### 4. 用户确认拆分方案

在开始正式写第 1 篇之前，先把拆分方案告诉用户。

确认内容：
- 拆成几篇是否合理
- 顺序是否合理
- 有没有哪一篇要优先写

### 5. 按篇写作，而不是一次写完整个系列

每次只处理一篇：
- 起标题方向
- 写正文
- 做 3 轮内部润色
- 如果要发公众号，再补封面和配图

不要在同一轮里同时推进多篇正文。

### 6. 每完成一篇就回写计划文件

完成一篇后必须更新计划文件：
- 勾掉总进度里的对应项
- 把该篇状态改为已完成
- 补上文章文件路径
- 补上草稿状态
- 补上发布状态

如果内容方向变了，也要同步改计划文件，而不是只在对话里口头说明。

### 7. 第二天继续时先读计划文件

跨天继续时，不要先靠记忆开写。

应先打开：
- 当前系列计划文件

确认：
- 已完成到哪一篇
- 下一篇是什么
- 有没有发布状态需要跟进

## 文章执行规则

每一篇系列文章都遵守：
- 先讲一个问题，不要试图一篇讲完整个系统
- 教学口吻优先，不写成“视频总结”口吻
- 如果用户希望独立成文，就不在正文里强调“来自某个视频”
- 文章内部要能独立阅读，不依赖读者先看过前一篇

## 进度记录规则

推荐状态：
- `未开始`
- `进行中`
- `已完成`
- `已上传草稿`
- `已发布`

如果一篇已经发出，但后面又要重写，不要删除记录，改成：
- `已发布（待修订）`

## 最小模板

每次新建系列计划文件，至少要有这几个部分：

```md
# <主题>拆分计划

## 来源

## 系列目标

## 总进度
- [ ] 第 1 篇
- [ ] 第 2 篇
- [ ] 第 3 篇

## 第 1 篇
状态：
- [ ] 未开始

暂定主题：

这篇要回答的问题：

核心内容：

建议标题方向：

完成后补充：
- 文章文件：
- 是否已上传草稿：
- 是否已发布：
```

## 一句话原则

先拆计划，再逐篇完成；每完成一篇，就把计划文件当成唯一真实进度来源。
